# 公司调研

基于公开信息对指定公司进行全维度系统性调研，覆盖基本面、业务产品、竞争格局、经营财务、风险动态等关键维度，最终输出一份钉钉在线文档形式的公司调研报告。

## 涉及产品

| 产品 | 用途 | 对应命令 |
|------|------|---------|
| 文档 (doc) | 创建调研报告文档、写入报告内容 | `dws doc create / update` |

## 前置依赖

- 需要 `dws`（dingtalk-workspace CLI）已安装并登录

## 能力清单

| 动作 | 命令/工具 | 必填参数 | 安全等级 | 说明 |
|------|----------|----------|---------|------|
| 网络搜索 | `web_search` | query | 只读 | 搜索公司公开信息 |
| 网页抓取 | `web_fetch` | url | 只读 | 获取公司官网、财报等页面全文 |
| 创建报告文档 | `dws doc create` | --name | 写入 | 创建调研报告钉钉文档 |
| 写入报告内容 | `dws doc update` | --node, --markdown, --mode | 写入 | 向报告文档写入 Markdown 内容 |

## 操作指引

### 输出形式
1. MUST: 触发本技能意味着用户需要一份正式的钉钉文档报告，禁止仅在对话中输出文本作为交付
2. MUST: 素材整理完毕后，必须通过 `task`（subagent）委派 Writer Agent 完成钉钉文档的创建与写入
3. MUST: 执行任何 `dws` 命令时均须附加 `--format json`，确保返回结构化可解析结果

### 信息收集
1. MUST: 所有数据和结论均须可溯源，无法确认的信息应标注"待补充验证"
2. MUST: 严禁虚构数据、伪造链接或捏造信息源；所有引用均需提供可验证的出处
3. MUST: 财务相关数据须注明数据来源及所属报告期（年度/季度）
4. MUST: 百分比类数据（如市场份额）须做加总校验，同口径份额之和不得超过 100%
5. MUST: 非上市/未独立融资的公司，须在对应章节显式说明原因（如"作为母公司全资子公司，无独立融资记录"），不得直接跳过
6. SHOULD: 关键财务指标应通过至少两个独立渠道交叉核实，偏差超过 10% 须在报告中说明
7. NOTE: 当用户限定了时间窗口时，须严格过滤超出该范围的资料
8. NOTE: 不同来源数据存在矛盾时，以公司官方披露为第一优先级，权威第三方为第二优先级，并在报告中标注差异
9. TRAP: 单轮搜索往往覆盖不全，每个维度须执行 2-3 轮递进式搜索以确保信息完整

### 报告生成
1. MUST: 通过 `dws doc create --name "<报告标题>" --format json` 新建文档，并从返回值中提取 `nodeId`
2. SHOULD: 报告内容较长时采用分段写入策略——首段使用 `--mode overwrite`，后续段落使用 `--mode append`
3. NOTE: `--markdown` 参数内的换行须为真实换行符（Unicode U+000A），而非转义字符串 `\n`
4. TRAP: `--mode overwrite` 会覆盖已有内容，后续追加务必切换为 `--mode append`

### 可读性规则
1. MUST: 三个及以上的并列条目应优先采用列表或表格呈现
2. MUST: 单个段落超过 150 字时须拆分为列表、子章节或高亮块
3. SHOULD: 表格内数值列右对齐，文本列左对齐
4. SHOULD: 时间序列信息采用有序列表，按时间由近及远排列

## 工作流

### 阶段 1: 确认公司主体
1. MUST: 从用户话中提取公司名称（含别名、英文名、股票代码等）
2. 明确研究边界：母公司/子公司/关联公司的范围
3. 确认可选参数：时间范围（默认近 1-3 年）、关注维度（默认全维度）、对标公司、研究深度
4. 校验: 公司名称明确
5. 失败: 公司名称模糊 → 询问用户明确

### 阶段 2: 信息收集（6 个维度）
依赖: 阶段 1 的公司名称和参数

按 A1-A6 六个维度依次收集，每个维度 2-3 轮 `web_search` + `web_fetch` 深度获取，所有素材写入本地 `draft.md`（append 模式）。

| 维度 | 目标 | 关键搜索方向 |
|------|------|-------------|
| A1. 公司基本面 | 基本档案信息 | 成立时间/总部/创始人/融资/上市/团队 |
| A2. 业务与产品 | 业务版图和收入来源 | 产品线/营收占比/商业模式/用户量 |
| A3. 竞争位置 | 行业地位和竞品对比（≥3 家竞品） | 行业排名/市场份额/竞品对比/壁垒 |
| A4. 经营数据 | 财务指标和经营效率 | 财报/ROE/研发投入/趋势分析 |
| A5. 最新动态 | 近期重大事件 | 新闻/战略调整/人事变动/合作并购 |
| A6. 补充资料（可选） | 钉钉文档已有研究 | `dws doc search` 搜索内部资料 |

各维度的详细搜索查询策略见 [references/search-strategy.md](./references/search-strategy.md)

### 阶段 3: 结构化分析
依赖: 阶段 2 的 `draft.md` 完整素材

基于收集的全部信息，依次完成以下分析并追加写入 `draft.md`：

1. **SWOT 分析**：每个象限至少 2 条有依据的要点，附带"依据"字段
2. **竞争壁垒分析**：从网络效应、规模经济、品牌价值、技术专利、转换成本、数据资产、生态体系等维度评估，给出综合护城河评级
3. **风险评估**：按市场/竞争/技术/监管/财务/运营/人才/地缘政治 8 类风险逐一评估，标注概率和影响程度
4. **综合评价**：6 维度评分（行业地位/成长性/盈利能力/竞争壁垒/管理团队/风险水平），采用 1-10 分制 + 核心结论

校验: 所有分析结论有数据/事实支撑
失败: 某模块数据不足 → 标注"数据不足，仅供参考"

### 阶段 4: 生成报告文档
依赖: 阶段 3 完成后的 `draft.md`

通过 `task`（subagent）委托 Writer Agent 执行以下步骤：

1. 读取本地 `draft.md` 获取全部调研素材和分析结论
2. MUST: 读取 [references/report-template.md](./references/report-template.md) 模板，严格使用模板定义的章节名（核心洞察/企业画像/商业版图/竞争力分析/财务健康度/风险雷达/近期信号/投研结论）、表格列名和评分体系（1-10 分制），禁止自行更改章节名或使用星级制
3. 调用 `dws doc create --name "{公司名称} 深度研报" --format json`，提取返回的 `nodeId`
4. 调用 `dws doc update --node <NODE_ID> --markdown "<报告内容>" --mode overwrite --format json`
5. 内容过长时分段：先 overwrite 第一部分，后续 append 追加
6. 校验: update 命令返回成功；逐章节核对是否与模板章节名一致
7. 失败: 写入失败 → 重试 1 次，仍失败则将报告内容保存为本地 Markdown 文件并告知用户

### 阶段 5: 交付
依赖: 阶段 4 的文档 URL

1. 向用户展示报告概要（一段话，150 字以内）
2. 附上钉钉在线文档链接
3. 禁止将报告全文输出到对话中

## 上下文传递表

| 阶段 | 从返回中提取 | 用于 |
|------|-------------|------|
| 阶段 2: `web_search` | 搜索结果摘要和链接 | `web_fetch` 的 url 参数 |
| 阶段 2: `web_fetch` | 页面详细内容 | 写入 `draft.md` 的素材 |
| 阶段 2: `doc search` | 文档 `nodeId` | `doc read` 的 `--node` 参数 |
| 阶段 4: `doc create` | `nodeId` | `doc update` 的 `--node` 参数 |
| 阶段 4: `doc create` | 文档 URL | 阶段 5 交付给用户 |

## 意图映射与易混淆场景

**触发本技能**："帮我调研一下 XX 公司"、"做一份公司调研报告"、"XX 公司怎么样？帮我全面分析一下"、"公司画像"、"我想了解 XX 公司的详细情况"

**不应触发**（路由到其他能力）：单篇文档创建/编辑、日常文档搜索、行业研究（非单一公司）、个人简历分析

## 使用示例

### 示例 1: 标准公司调研
**用户说**: "帮我调研一下字节跳动"
**执行步骤**:
1. 确认公司主体为"字节跳动"，默认全维度、近 1-3 年数据
2. 按 A1-A6 六个维度依次进行信息收集，每个维度 2-3 轮 `web_search` + `web_fetch` 深度获取
3. 所有素材写入本地 `draft.md`
4. 完成 SWOT、竞争壁垒、风险评估、综合评价等结构化分析
5. 委托 Writer Agent：读取 `draft.md`，调用 `dws doc create --name "字节跳动 公司调研报告" --format json`
6. 调用 `dws doc update --node <NODE_ID> --markdown "<报告内容>" --mode overwrite --format json`
**期望输出**: "已生成字节跳动公司调研报告，覆盖公司概况、业务产品、竞争格局等六大维度。[查看报告](文档URL)"

## 错误处理

| 阶段 | 错误 | 处理 | 是否终止流程 |
|------|------|------|-------------|
| 阶段 1 | 公司名称无法识别 | 询问用户明确公司名称 | 否，等待用户回复 |
| 阶段 2 | `web_search` 某维度无结果 | 换关键词重试 1 次，仍无结果则标注"数据不足" | 否 |
| 阶段 2 | `web_fetch` 页面无法访问 | 跳过该页面，使用已有搜索结果 | 否 |
| 阶段 2 | `dws doc search` 无内部资料 | 跳过 A6，标注"未找到内部已有研究" | 否 |
| 阶段 3 | 某分析模块数据不足 | 标注"数据不足，仅供参考" | 否 |
| 阶段 4 | `dws doc create` 失败 | 重试 1 次，仍失败则报告错误信息 | 是 |
| 阶段 4 | `dws doc update` 内容过长失败 | 分段写入（先 overwrite 第一部分，后续 append） | 否，回退到分段写入 |
| 阶段 4 | `dws doc update` 权限不足 | 提示用户检查文档权限 | 是 |
| 全局 | `dws` 命令不可用（未安装或未登录） | 提示用户确认 dws 已安装，并重新执行上一条命令（最多重试两次） | 是 |
