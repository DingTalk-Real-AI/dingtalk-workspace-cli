# 论文深度解读质量检查清单

> 使用方式：在输出报告前，逐项核对。所有必检项必须全部通过，才可执行 `dws doc create`。

---

## 一、报告完整性门禁

### Phase 1-2：信息采集阶段

- [ ] 核心论文原文 PDF 已获取并完整阅读（摘要 + 引言 + 方法 + 定理 + 实验 + 局限性），**禁止仅凭摘要或博客二手解读**
- [ ] 核心论文版本已确认（arXiv 版本号 / camera-ready 版本），版本差异已记录
- [ ] 话题背景：已定位 3-8 篇奠基性工作，领域脉络已建立
- [ ] 近期工作：时间窗口严格限定在 24-36 个月内，超出范围的论文仅在研究脉络章节引用
- [ ] 每篇纳入的近期论文均已阅读原文（至少：摘要 + 方法 + 主要结果 + 实验），并输出结构化卡片

### Phase 3：报告撰写阶段

**§1 核心论文解构**
- [ ] 问题建模：canonical objective 已用 LaTeX 公式写出，所有符号首次出现时定义，全文符号严格统一
- [ ] 关键假设：所有影响定理成立的前提条件已列出，并说明"作用"和"能否放松"
- [ ] 推导路线：从原始问题到可解形式的完整路径已写出（哪怕是简化的 proof sketch）
- [ ] 核心公式：至少 3 个关键公式，每个公式逐项解释，无裸公式堆砌
- [ ] 理论保证：所有定理使用"前提条件 → 结论 → 实践意义 → 证明要点"四段式，不得省略任何一段
- [ ] 算法实现：迭代更新式已写出，计算复杂度已标注，实现注意事项已说明
- [ ] 实验可信度：5 个维度均已评级（⭐-⭐⭐⭐⭐⭐），评级有具体理由，**不得默认满星**

**§2 研究脉络**
- [ ] 领域演进时间线：至少覆盖早期 / 中期 / 近期三个阶段，每个阶段有代表工作和遗留局限
- [ ] 统一问题形式：领域 canonical 问题已用统一符号写出，主流研究范式已列出

**§3 前沿进展对比**
- [ ] 研究分支分类：先建立分类框架（分支名称反映技术路线，非泛泛"理论/应用"），再填写对比矩阵
- [ ] 横向对比矩阵：所有论文使用相同列头，复杂度列使用统一符号，横向可比
- [ ] 进展关系分析：每篇纳入论文均有明确论证句（须含技术细节，禁止仅罗列摘要），至少 3 条

**§4 批判性评估**（新增，aily 版本无此章节）
- [ ] 核心论文局限性：理论 / 实验 / 工程 / 适用范围四类均已评估，每条局限具体可验证
- [ ] 严重程度评级已填写（🔴 高 / 🟡 中 / 🟢 低），不得全部标为低风险
- [ ] 领域争议点：若存在争议已列出；若无争议已明确标注"当前领域主要结论已达成共识"
- [ ] 可信度综合评级：4 个维度均已评级，评级有具体理由

**§5 开放问题**
- [ ] 近期可攻克问题（1-2 年）：至少 2 条，每条含难度标注和具体可验证描述
- [ ] 中长期挑战（3-5 年）：至少 2 条，每条含难度标注
- [ ] 潜在突破口：至少 1 条，含判断理由和关键前提条件

**附录**
- [ ] 统一符号表：汇总全文所有符号，含类型、含义、首次出现章节
- [ ] 参考文献：所有引用含作者 + 完整标题 + 会议/期刊全称 + 年份 + 链接
- [ ] 复现指南：数据集 / 官方代码 / 关键超参数 / 随机种子 / 硬件需求 / 已知复现难点均已填写

### Phase 4：文档创建阶段

- [ ] 报告内容已写入本地临时文件（`/tmp/paper-report.md`）
- [ ] `dws doc create` 已执行，nodeId 已提取
- [ ] `dws doc update --markdown-file` 已执行，文档写入成功
- [ ] 钉钉文档链接已返回给用户，并附 2-3 条核心发现摘要

---

## 二、报告质量门禁

> 以下任一项不满足，视为「浅层报告」，必须补齐后再输出，不得直接创建文档。

| 质量维度 | 最低要求 | 检查方法 |
|---------|---------|---------|
| 公式覆盖 | 至少 3 个关键公式（canonical form / 核心方法 / 定理） | 数公式数量 |
| 推导深度 | 至少 1 条完整推导路线（proof sketch 可接受） | 检查 §1.2 |
| 对比广度 | 至少 1 张横向对比矩阵（维度统一，横向可比） | 检查 §3.2 |
| 论证质量 | 至少 3 条关系论证句（含技术细节，非摘要罗列） | 检查 §3.3 |
| 批判深度 | 至少 1 条具体可验证的局限性（非空泛表述） | 检查 §4.1 |
| 结论可验证性 | 无空泛形容词，所有结论可指向公式/实验数据 | 全文扫描 |
| 开放问题质量 | 至少 3 条可验证的后续研究问题 | 检查 §5 |

## 三、成功 / 失败判定

**✅ 成功**：
- 报告覆盖 §0-§5 全部章节，附录完整
- 满足全部质量门禁（7 项）
- 钉钉文档链接已返回给用户

**❌ 失败**（须立即修复，不得直接结束任务）：
- 仅摘要级理解，未读完整 PDF → 回到 Phase 1 重新获取原文
- 缺少批判性评估章节（§4）→ 补充局限性分析和可信度评级
- 关系论证句少于 3 条，或仅罗列摘要无技术细节 → 补充论证
- 引用缺少完整出处（作者 + 标题 + 会议/期刊 + 年份）→ 补全引用
- 文档创建失败且未告知用户 → 将报告内容输出到对话中

## 四、系统类错误处理

| 错误类型 | 处理方式 |
|---------|---------|
| PDF 获取失败（WebFetch 超时 / 403） | 依次尝试：arXiv HTML 版 → Semantic Scholar → ACL Anthology → OpenReview；仍失败则标注「原文获取失败，以下分析基于摘要，关键定理可信度受限」 |
| arXiv 访问受限 | 尝试 `ar5iv.org`（HTML 渲染版）或 Semantic Scholar PDF 链接 |
| 论文不存在 / 标题有误 | 告知用户，请求提供正确的 arXiv ID 或 DOI，**不得继续分析** |
| 近期工作搜索无结果 | 扩大时间窗口至 48 个月，或换用英文关键词 / 同义词重试；仍无结果则在报告中标注「近期工作数据不足，对比分析受限」 |
| `dws doc create` 失败 | 检查 dws 连接状态，重试一次；仍失败则将完整报告内容输出到对话中，提示用户手动创建文档 |
| `dws doc update` 失败 | 检查 nodeId 是否正确，重试 `--mode overwrite`；仍失败则分段输出报告内容 |
