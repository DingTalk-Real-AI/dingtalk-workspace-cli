# 论文深度解读搜索策略与分析方法

> 本文件对应 SKILL.md 中 Step 1-5 的具体执行指南，与 output-template.md 的章节结构一一对应。

---

## Step 1：定位并获取核心论文

**搜索命令**（按优先级依次尝试）：

```
# 精确定位
web_search: "{论文完整标题}" arXiv
web_search: "{论文标题}" "{第一作者姓}" {年份} PDF

# 备用定位（标题不确定时）
web_search: "{核心方法名}" "{第一作者姓}" {会议名} {年份}
web_search: "{arXiv ID}"
```

**获取原文**（按优先级）：

```
WebFetch: https://arxiv.org/abs/{arXiv ID}        # 确认版本号
WebFetch: https://arxiv.org/pdf/{arXiv ID}        # 获取 PDF 全文
WebFetch: https://ar5iv.org/html/{arXiv ID}       # 备用：HTML 渲染版（PDF 失败时）
WebFetch: https://api.semanticscholar.org/...     # 备用：Semantic Scholar
```

**必须阅读的章节**（不得跳过）：
- 摘要（Abstract）
- 引言（Introduction）：确认问题设定和主要贡献
- 方法（Method / Approach）：完整阅读
- 主要定理 / 关键引理（Theorem / Lemma）：含假设和证明
- 实验（Experiments）：Baseline 选取、指标、消融
- 局限性（Limitations / Discussion）

**版本核对**：记录 arXiv 版本号（v1/v2/...），确认是否有 camera-ready 版本，两版本有差异时以 camera-ready 为准。

## Step 2：核心论文结构化分析

> 对应 output-template.md §1（核心论文解构）

### 2.1 问题建模

- 写出统一符号体系下的 canonical objective（LaTeX 格式）
- 建立符号表：每个符号首次出现时定义，全文严格统一，不得在不同章节使用不同符号表示同一概念
- 列出所有关键假设，并说明：该假设保证了什么 / 实际中是否容易满足 / 能否放松及代价

### 2.2 方法与推导

- 写出从原始问题到可解形式的完整推导路线，例如：
  ```
  原始 minimax 问题 → Lagrangian 对偶 → 正则化松弛 → 随机梯度可解形式
  ```
- 识别定义该方法的 1-3 个核心公式，逐项解释每一项的含义
- 标注推导中的关键技巧（如：强对偶性成立的条件、正则化参数的选取依据）

### 2.3 理论结果（定理化呈现）

每个关键定理严格按以下四段式写出，不得省略任何一段：

```
前提条件：假设 A1, A2, A3 成立，且步长满足 η_t = O(1/√t)
结论：E[f(x^T) - f*] ≤ O(1/√T)
实践意义：迭代 T 次后，期望误差以 1/√T 速率收敛，与样本量 n 无关
证明要点：关键不等式 E[‖x^{t+1} - x*‖²] ≤ E[‖x^t - x*‖²] - 2η_t(f(x^t) - f*) + η_t²G²
```

### 2.4 算法细节

- 写出迭代更新式（LaTeX 格式），标注每个变量的含义
- 给出计算复杂度 $O(\cdot)$，说明瓶颈在哪一步
- 标注实现注意事项：超参数敏感性 / 数值稳定性 / 并行化可行性

### 2.5 实验可信度评估

对以下 5 个维度逐一评估，给出 ⭐-⭐⭐⭐⭐⭐ 评级和具体理由：

| 维度 | 评估要点 |
|------|---------|
| Baseline 选取合理性 | 是否包含最新 SOTA 和经典方法？有无明显遗漏？ |
| 指标与主张的匹配度 | 所选指标是否能直接支撑论文核心主张？有无指标选择偏差？ |
| 消融实验完整性 | 有/无？覆盖了哪些关键组件？是否遗漏重要变量？ |
| 统计显著性检验 | 有/无置信区间或 p-value？多次随机种子是否报告？ |
| 数据集多样性 | 数据集数量和覆盖场景，是否存在数据集选择偏差？ |

## Step 3：研究脉络调研

> 对应 output-template.md §2（研究脉络）

**搜索命令**：

```
# 综述与教材级引用
web_search: "{topic} survey review 综述"
web_search: "{topic} tutorial introduction overview"
web_search: "{topic} seminal work foundational paper"

# 领域演进
web_search: "{topic} history evolution milestone"
web_search: "{核心方法前身} original paper {奠基作者}"
```

**产出要求**：
- 识别 3-8 篇奠基性工作，按时间线排列（早期 / 中期 / 近期）
- 每个时期标注：代表工作 + 核心贡献 + 遗留给后续工作的局限
- 建立领域 canonical 问题的统一符号形式，后续所有章节严格遵循

## Step 4：前沿进展调研

> 对应 output-template.md §3（前沿进展对比）

**时间窗口**：默认近 24-36 个月。超出范围的论文仅在 §2 研究脉络中引用，不得纳入 §3 对比。

**搜索命令**（并行执行，覆盖多个检索角度）：

```
# 关键词 + 时间限定
web_search: "{topic} {核心方法名} 2024 2025 paper"
web_search: "{topic} {顶会名，如 NeurIPS ICML ICLR} 2024 2025"

# 引用追踪
web_search: "{核心论文标题}" cited by 2024 2025
web_search: site:arxiv.org "{核心论文第一作者}" {topic} 2024

# 同义词扩展
web_search: "{核心术语同义词1} {核心术语同义词2} new method 2024"
web_search: "{topic} {相关子领域} recent advances"

# 批判性视角（用于 §4 批判性评估）
web_search: "{核心论文标题}" limitation weakness critique
web_search: "{topic} open problem challenge unsolved"
```

**每篇候选论文的处理流程**：

1. **快速筛选**：获取摘要，判断是否与核心论文共享同一建模对象或在相同 benchmark 上有直接对比
2. **深度阅读**：WebFetch PDF，必须阅读：摘要 + 方法 + 主要定理/结果 + 实验
3. **输出结构化卡片**：

```
论文：{标题}（{作者}, {会议/期刊}, {年份}）
问题设定：{目标函数类型 + 关键假设，与核心论文使用统一符号}
核心方法：{方法名或关键公式/界}
计算复杂度：O(...)
实验规模：{数据集名称 / 参数规模}
与核心论文的关系：{改进/替代/互补/否定 — 须含技术细节}
```

**质量控制**：
- 禁止仅凭博客、知乎、Medium 等二手解读下结论，必须阅读原文
- 不同论文结论冲突时，记录冲突点（假设不同？数据集不同？指标不同？实现差异？）并在 §4.2 领域争议点中列出

## Step 5：结构化分析

> 对应 output-template.md §3.1-3.3 + §4

### 5.1 研究分支分类

先建立分类框架，分支名称须反映核心技术路线（非泛泛的"理论/应用"），例如：

| 分支 | 核心技术路线 | 典型代表 |
|------|-----------|---------|
| **收敛速率强化** | 方差缩减 + 动量加速 | {论文名} |
| **假设条件放松** | 非凸扩展 / Hölder 连续 | {论文名} |
| **计算效率提升** | 随机化 / 分布式 / 近似求解 | {论文名} |
| **下游应用拓展** | 迁移到 NLP / CV / 强化学习场景 | {论文名} |

### 5.2 横向对比矩阵

所有论文使用相同列头，确保横向可比：

```
列头：论文（年份） | 分支 | 问题设定与假设 | 核心方法/关键界 | 复杂度 | 实验规模 | 与核心论文的关系
```

复杂度列须使用统一符号（与 §1.4 一致）。

### 5.3 进展关系分析

对每篇纳入论文写出明确的论证句，须含技术细节，禁止仅罗列摘要。

**论证句写作规范**（三种典型模式）：

- **改进型**：「{论文R}将核心论文的{具体假设/方法}替换为{新假设/方法}，在{具体条件}下将{指标}从{旧值}改进至{新值}，代价是{具体代价}。」
- **放松型**：「{论文R}在核心论文的假设{Ax}（{假设内容}）上做了放松，改为{新假设}，收敛速率从{旧界}退化至{新界}，但适用范围扩展至{新场景}。」
- **辨析型**：「{论文R}在相同 benchmark 上报告了更高{指标}（+{数值}），但其优势主要来自{具体原因，如：更大的预训练模型容量（参数量{N}×）/ 更多的训练算力 / 不同的数据预处理}，而非算法本身的改进；在控制{变量}后，两者差异不显著（{具体数值}）。」

### 5.4 批判性评估要点

> 对应 output-template.md §4，独立于论文作者立场进行批判

- **理论局限**：哪个假设过强？在什么实际场景下会违反？有无后续工作解决？
- **实验局限**：数据集是否单一？是否缺乏消融？是否未与最新 SOTA 对比？
- **工程局限**：计算开销是否可接受？超参数是否敏感？能否在工业场景落地？
- **适用范围**：理论保证在什么条件下成立？非凸 / 大规模 / 分布式场景是否有效？
- **领域争议**：是否存在尚未达成共识的争议？持不同观点的代表工作是哪些？
