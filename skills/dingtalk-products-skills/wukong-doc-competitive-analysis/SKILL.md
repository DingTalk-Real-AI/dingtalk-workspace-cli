# 钉钉竞品分析 Skill

通过公开信息采集与结构化分析，生成覆盖产品定位、功能对比、商业模式、市场格局、用户口碑五大维度的竞品研究报告，并输出为钉钉在线文档。

## 适用边界

- **Distinct from**：`wukong-doc-paper-deepresearch`（学术论文精读）、`market-report`（行业宏观趋势，无具体竞品横向对比）
- **Do NOT use for**：单一产品功能介绍、用户手册、不含竞品对比的用户研究

## 工具权限

| 工具 | 用途 | 约束 |
|------|------|------|
| `web_search` | 竞品公开信息检索 | 仅访问公开页面 |
| `WebFetch` | 官网定价页、产品页直采 | 禁止访问需登录的内容 |
| `dws doc create` | 新建钉钉文档 | 每次任务只新建一个文档 |
| `dws doc update` | 写入报告内容 | 仅操作本次 create 返回的 nodeId |

所有 `dws` 命令必须附加 `--format json`。

## 触发意图识别

| 用户表达 | 分析重点 | 输出形式 |
|---------|---------|---------|
| "做竞品分析" / "分析竞争对手" | 五维度全量分析 | 完整报告 |
| "对比 XX 和 YY 的功能" | 功能矩阵 + 体验对比 | 聚焦功能章节 |
| "XX 的定价策略怎么样" | 商业模式 + 定价对比 | 聚焦商业章节 |
| "我们和竞品的差距在哪" | 差距识别 + 机会点 | 聚焦战略建议章节 |
| "写一份竞品报告" | 五维度全量分析 | 完整报告，需先确认目标产品 |

## 易混淆场景路由

| 用户说 | 判断线索 | 正确路由 |
|--------|---------|---------|
| "帮我做行业分析报告" | 关注宏观趋势、行业规模，无具体竞品横向对比 | 路由到 market-report |
| "帮我精读这篇论文" | 含"论文"/"paper"/"arXiv"/"研究报告" | 路由到 wukong-doc-paper-deepresearch |
| "写一份产品介绍文档" | 介绍自身产品，无竞品对比需求 | 普通文档生成 |
| "分析用户满意度" | 聚焦 NPS/用户评价，无竞品横向对比 | 用户研究类 Skill |

## 典型场景示例

"**示例 1：全量竞品分析**
- 用户：帮我做一份悟空 vs 小龙虾 vs Manus 的竞品分析
- 执行：确认分析范围 → 并行采集三个产品的公开数据 → 完成五维度分析 → 创建钉钉文档写入报告
- 输出：含执行摘要、功能矩阵、商业模式对比、战略建议的完整钉钉文档链接

**示例 2：聚焦战略差距**
- 用户：悟空和 Manus 的核心差距在哪，有什么机会点
- 执行：确认目标产品 → 采集 Manus 公开信息 → 重点输出差距分析 + 机会点识别
- 输出：聚焦差距与行动建议的精简版钉钉文档链接"

## 输入参数

| 参数 | 是否必填 | 说明 |
|------|---------|------|
| 目标产品名称 | 必填 | 分析的主体产品 |
| 竞品列表 | 可选 | 默认自动识别 3-5 个主要竞品 |
| 重点分析维度 | 可选 | 默认覆盖全部五个维度 |
| 报告受众 | 可选 | 影响报告深度和侧重点（产品团队/管理层/投资人） |

## 执行流程

### Phase 1：范围确认

在开始研究前，与用户确认：
- 目标产品及其核心使用场景
- 竞品范围（直接竞品 / 间接竞品 / 潜在威胁）
- 报告受众与核心关注点

> 范围确认是必须步骤，不得跳过。

### Phase 2：并行信息采集

对目标产品和每个竞品**并行**执行搜索采集，覆盖以下方向：

- **产品能力**：官网功能页、产品更新日志、第三方评测
- **定价信息**：必须 WebFetch 官网定价页直采，禁止使用第三方摘要
- **市场表现**：用户规模、增速、融资/估值等公开数据
- **用户口碑**：G2、Capterra、知乎、App Store 评价等多平台交叉验证
- **技术能力**（如适用）：部署方式、API、安全认证、集成数量

详细搜索命令模板见 [search-patterns](references/search-patterns.md)。

### Phase 3：结构化分析

信息采集完成后，依次完成以下分析模块：

| 模块 | 产出内容 |
|------|---------|
| A. 竞争格局 | 竞品分类（直接/间接/潜在）+ 市场定位象限描述 |
| B. 竞品深度画像 | 每个竞品的定位、优势、短板、最新动态 |
| C. 横向能力对比 | 功能矩阵（统一四档）+ 体验评分 + 技术能力（可选） |
| D. 商业模式分析 | 定价策略、计费逻辑、增长飞轮对比 |
| E. 战略机会识别 | 差距分析、机会点（含优先级）、行动建议 |

### Phase 4：创建钉钉文档

**Step 4a**：创建文档
```bash
dws doc create --name "竞品研究：{目标产品} 竞争格局分析 {YYYY-MM}" --format json
```
从返回结果中提取 `nodeId`，用于后续写入。

**Step 4b**：将报告写入临时文件后通过文件路径写入文档

```bash
dws doc update --node <nodeId> --markdown-file /tmp/competitive-report.md --mode overwrite --format json
```

返回文档链接给用户，并附上 2-3 条核心发现摘要。

## 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| WebFetch 超时 / 403 | 换用 `web_search` 获取摘要，标注「直采失败，数据来自搜索摘要，建议人工核实」 |
| 竞品信息搜索无结果 | 尝试英文名 / 全称 / 缩写扩展搜索；仍无结果则标注「公开信息不足」 |
| 官网无定价页 | 标注「定价需联系销售」，禁止编造或引用第三方估算 |
| `dws doc create` 失败 | 重试一次；仍失败则将完整报告输出到对话中，提示用户手动创建文档 |
| `dws doc update` 失败 | 检查 nodeId 是否正确，重试 `--mode overwrite`；仍失败则分段输出报告内容 |

## NEVER DO

- **禁止编造数据**：无法获取的数据必须标注「需补充」
- **禁止非官网定价**：定价必须从官网直采，不得引用博客、评测、第三方摘要
- **禁止无数据下结论**：所有竞争判断必须有来源支撑
- **禁止访问需登录内容**：仅采集公开可访问的信息
- **禁止操作已有文档**：只能写入本次 `create` 返回的 nodeId 对应文档
- **禁止跳过范围确认**：Phase 1 是必须步骤

## 常见失败模式

| 失败表现 | 根因 | 正确做法 |
|---------|------|---------|
| 各产品对比维度不统一 | 未提前锁定分析框架 | Phase 1 先确定维度清单，所有产品用同一清单逐项比对 |
| 定价数据不准确 | 引用第三方摘要或过期数据 | 必须 WebFetch 官网定价页，标注获取日期 |
| 报告只有事实罗列无分析 | 跳过 Phase 3 结构化分析 | 完成全部 A-E 分析模块后再写文档 |
| 行动建议空洞 | 缺少具体策略和时间节点 | 每条建议必须含：具体行动 + 可衡量目标 + 时间窗口 + 优先级 |
| 文档写入失败后直接终止 | 未做降级处理 | 重试一次；仍失败则将报告内容输出到对话中 |

质量检查清单见 [quality-checklist](references/quality-checklist.md)。
