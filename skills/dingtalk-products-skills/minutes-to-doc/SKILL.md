# 听记内容同步到文档

将听记中的纪要内容智能同步到已有钉钉文档：分析原文档结构和格式，将新内容融入到合适位置（优先补充到现有章节），实现"会议讨论 → 文档沉淀"的无缝衔接。

## 核心价值

**讨论不落空，格式不割裂**：会议讨论结果智能融入现有文档结构，而非简单追加一堆"会议补充"，让文档保持整洁统一。

## 交互风格

**像助手，不像机器人**：不暴露技术细节（不说"API"、"taskUuid"、"nodeId"），直接做、自然说。

## 严格禁止 (NEVER DO)

- **禁止暴露技术细节**：不说"调用命令"、"taskUuid"、"nodeId"等术语
- **禁止编造内容**：所有同步的内容必须来自听记原文，不可自行添加
- **禁止跳过确认**：更新方案必须经用户确认后才能执行
- **禁止猜测 ID**：听记 ID 和文档 ID 必须从 URL 提取或从 list 返回中获取
- **禁止未经确认直接修改**：任何对原文档的修改/删除都需要用户明确确认

## 更新模式

用户在确认阶段需选择更新模式：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **新建文档** | 创建新文档，原文档不变 | 想保留原文档作为备份 |
| **修改原文档** | 直接在原文档上增删改 | 希望文档保持最新状态 |

| 产品 | 安全等级 | 用途 | 对应命令 |
|------|----------|------|---------|

## 输入

**必填**：听记 URL 或描述 + 目标文档 URL 或描述

**可选**：
- 同步范围（全部纪要 / 仅结论 / 仅待办）
- 插入位置偏好（文档末尾 / 指定章节后）
- 新增板块标题

## 执行流程

### Step 1：定位听记

| 输入方式 | 操作 |
|---------|------|
| 用户传入听记 URL | 从 URL 提取 `taskUuid`，调用 `dws minutes get info --id <taskUuid> --format json` |
| 自然语言描述 | 调用 `dws minutes list mine --max 20 --format json`，按时间/主题匹配 |

**MUST**：多候选时列出候选列表，询问用户确认后再继续。

### Step 2：获取听记内容

```bash
dws minutes get summary --id <taskUuid> --format json
dws minutes get transcription --id <taskUuid> --format json
```

提取核心内容：会议主题、关键结论、讨论要点、决策事项。

### Step 3：展示听记内容并确认

向用户展示提取到的纪要内容摘要（如"从「XX会议」中提取了以下要点..."），**MUST**：必须等用户确认这些内容后才能继续。

### Step 4：定位目标文档

| 输入方式 | 操作 |
|---------|------|
| 用户传入文档 URL | 从 URL 提取 `nodeId`，调用 `dws doc info --node <nodeId> --format json` |
| 自然语言描述 | 调用 `dws doc search --query "<关键词>" --format json`，匹配文档 |

**MUST**：多候选时列出候选列表，询问用户确认后再继续。

### Step 5：分析文档结构并匹配内容

```bash
dws doc read --node <nodeId> --format json
dws doc block list --node <nodeId> --format json
```

分析文档并制定更新方案：
1. **识别文档结构**：章节标题、列表、表格等格式
2. **匹配内容归属**：听记中的每条内容应该归入文档的哪个章节
3. **决定更新方式**：
   - **融入现有章节**：内容与某章节主题匹配 → 在该章节补充
   - **新增段落**：无匹配章节 → 在文档末尾新增

详细匹配规则见 [references/doc-structure-rules.md](./references/doc-structure-rules.md)。

### Step 6：生成更新方案并确认模式

向用户展示更新方案，说明哪些内容融入现有章节、哪些需新增、哪些原有内容可能被修改/替换。

**MUST**：让用户选择更新模式：
- **新建文档**：创建新文档，原文档保持不变
- **修改原文档**：直接在原文档上增删改

### Step 7：执行文档更新

根据用户选择的模式执行：
- **新建文档**：`doc create` 创建 → `block insert` 写入内容
- **修改原文档**：`block insert/update/delete` 按方案增删改

详见 [references/doc-structure-rules.md](./references/doc-structure-rules.md)。

### Step 8：交付结果

向用户展示完成信息（更新了哪些章节、新增了什么），包含文档链接。

## 上下文传递表

| 阶段 | 从返回中提取 | 用于 |
|------|-------------|------|
| Step 1 | `taskUuid`（从 URL 或 list 返回） | Step 2 的 `--id` 参数 |
| Step 2 | 摘要内容、会议主题 | Step 3 展示 + Step 6 生成内容 |
| Step 4 | `nodeId`（从 URL 或 search 返回） | Step 5/7 的 `--node` 参数 |
| Step 5 | 文档结构、`blockId` | Step 7 确定更新位置 |

## 踩坑提醒

| 错误做法 | 正确做法 | 原因 |
|---------|---------|------|
| 不问就直接改原文档 | 让用户选择"新建"或"修改原文档" | 用户可能想保留原文档 |
| 所有内容都新增段落 | 先分析原文档，能融入就融入 | 保持文档结构整洁 |
| 编造 taskUuid 或 nodeId | 从 URL 或 list 返回中获取 | 虚构 ID 会导致失败 |
| 融入/修改后格式混乱 | 保持与原文档格式一致 | 避免风格割裂 |

## 使用示例

**示例 1：修改原文档**
用户："把听记里技术方案讨论更新到方案文档" → AI："建议在「技术设计」章节补充。① 新建文档 ② 修改原文档？" → 用户："直接改" → 完成

**示例 2：新建文档**
用户："把会议讨论同步到项目文档" → AI："需要更新多处。① 新建文档 ② 修改原文档？" → 用户："新建" → 创建新文档

## 错误处理

| 阶段 | 错误 | 处理 |
|------|------|------|
| Step 1/4 | 听记/文档未找到 | 询问更准确描述或 URL |
| Step 5/7 | 权限不足 | 提示检查权限 |
| Step 7 | 写入失败 | 重试 1 次，仍失败则输出内容让用户手动处理 |
