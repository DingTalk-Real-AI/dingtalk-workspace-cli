# 听记报告生成器

根据钉钉听记内容生成结构化报告并输出为钉钉在线文档，支持多种报告类型和 HTML 可视化导出。

## 涉及产品

| 产品 | 用途 | L1 技能 |
|------|------|---------|
| 听记 (minutes) | 获取听记列表、摘要、转写、待办 | dws |
| 文档 (doc) | 创建报告文档、写入内容 | dws |

## 能力清单

| 动作 | 命令 | 安全等级 | 说明 |
|------|------|---------|------|
| 查询听记列表 | `dws minutes list mine` | 只读 | 获取 taskUuid |
| 获取听记摘要 | `dws minutes get summary --id <uuid>` | 只读 | AI 结构化摘要 |
| 获取转写原文 | `dws minutes get transcription --id <uuid>` | 只读 | 完整语音转写 |
| 获取待办事项 | `dws minutes get todos --id <uuid>` | 只读 | 听记中的待办 |
| 创建文档 | `dws doc create --name <title>` | 写入 | 创建报告文档 |
| 写入内容 | `dws doc update --node <id> --markdown <content>` | 写入 | 写入 Markdown |

| 用户说... | 线索 | 执行动作 |
|----------|------|---------|

| 用户说... | 容易误判为 | 正确路由 | 判断依据 |
|----------|----------|---------|---------|

## 操作指引

### 数据获取
1. MUST: 所有命令加 `--format json`
2. MUST: 从用户输入提取听记来源（URL / 关键词 / "最近 N 条"）
3. TRAP: `get summary` 返回 AI 摘要，不是原始转写

### 报告生成
1. MUST: 报告以钉钉文档交付，禁止将全文输出到对话
2. MUST: 使用 `doc create` 创建文档，提取 `nodeId`
3. TRAP: `--mode overwrite` 会清空原内容，追加用 `--mode append`
4. NOTE: `--markdown` 换行必须是真实换行符，不是 `\n` 字面量

## 工作流

### 阶段 1: 确认参数
1. 提取：听记来源（URL / 关键词 / "最近 N 条"）、用户材料
2. 判断报告类型：
   - IF 用户直接指定（"写个会议纪要"、"生成项目周报"）→ 使用指定类型，跳过确认
   - IF 用户未指定 → 记录待阶段 3 智能推断
3. IF 未明确听记 → 询问"您想整理哪些听记？"
4. 校验: 至少有一个明确听记来源
5. 失败: 列出最近 5 条听记供选择

### 阶段 2: 获取听记数据
依赖: 阶段 1 的听记来源

**用户提供 URL**: 从 `https://shanji.dingtalk.com/app/transcribes/<uuid>` 提取 taskUuid
1. `dws minutes get info --id <uuid> --format json` → 标题、时间
2. `dws minutes get summary --id <uuid> --format json` → AI 摘要
3. `dws minutes get todos --id <uuid> --format json` → 待办

**用户说"最近 N 条"**: 
1. `dws minutes list mine --max <N> --format json` → 获取列表，提取 taskUuid
2. 对每篇执行上述 1-3 步

校验: 每篇听记至少获取到摘要
失败: 某听记失败 → 跳过，继续处理其他

### 阶段 3: 内容整合与类型确认
依赖: 阶段 2 的听记数据 + 阶段 1 的用户材料

**报告类型确认**:
- IF 阶段 1 已确定类型 → 直接使用
- IF 未指定类型 → 分析听记内容，智能推断：
  - 客户拜访类：可能是「最佳实践案例」「需求清单」「媒体宣传稿」「拜访纪要」
  - 项目类：可能是「周报」「里程碑报告」「复盘总结」
  - 培训类：可能是「培训纪要」「知识沉淀」「FAQ 整理」
  - 向用户展示 2-3 个推荐选项，获得确认后继续

**内容整合**:
1. 提取核心结论、讨论要点、待办事项
2. IF 多篇 → 按时间线整合，冲突结论标注来源
3. IF 有用户背景 → 融入报告开头
4. 按确定的报告类型选择模板（见 [report-template.md](./references/report-template.md)）

### 阶段 4: 生成文档
依赖: 阶段 3 的整合内容
1. `dws doc create --name "{类型}: {主题} - {日期}" --format json` → 提取 nodeId
2. 组织 Markdown：封面 → 执行摘要 → 正文 → 待办 → 附录
3. `dws doc update --node <nodeId> --markdown "<内容>" --mode overwrite --format json`
4. IF 过长 → 分段：先 overwrite，后续 append

校验: update 返回成功
失败: 重试 1 次，仍失败则报告错误

### 阶段 5: 交付
1. 展示报告概要（100 字内）+ 文档链接
2. 询问："需要 HTML 可视化版本吗？可导出 PDF"

## 上下文传递表

| 阶段 | 提取 | 用于 |
|------|------|------|
| 阶段 2: `minutes list` | `taskUuid` | get 命令的 `--id` |
| 阶段 2: `minutes get info` | 标题、时间 | 报告封面 |
| 阶段 2: `minutes get summary` | 摘要 Markdown | 报告正文 |
| 阶段 4: `doc create` | `nodeId` | `doc update` 的 `--node` |
| 阶段 4: `doc create` | 文档 URL | 交付给用户 |

## 使用示例

### 示例 1: 用户明确指定类型（直接生成）
**用户说**: "把这个听记整理成会议纪要 https://shanji.dingtalk.com/app/transcribes/abc123"
**执行**:
1. 提取 taskUuid，类型已明确为「会议纪要」
2. 获取摘要 → 创建文档 → 写入
**输出**: "已生成会议纪要，[查看报告](URL)。需要 HTML 版本吗？"

### 示例 2: 用户未指定类型（智能推断）
**用户说**: "整理一下上次客户拜访的听记"
**执行**:
1. 获取听记摘要，分析内容：客户拜访、讨论合作方案、提到需求点
2. 推断可能的类型，向用户确认：
   > "根据听记内容，推荐以下报告类型：
   > 1. **客户拜访纪要** - 记录沟通要点和后续行动
   > 2. **需求清单** - 整理客户提出的需求
   > 3. **合作方案摘要** - 提炼方案要点
   > 请选择或告诉我您想要的类型"
3. 用户选择后生成对应报告
**输出**: "已生成客户拜访纪要，[查看报告](URL)"

### 示例 3: 多篇听记汇总
**用户说**: "整理本周的听记"
**执行**: list mine → 筛选本周 → 获取摘要 → 推断类型/确认 → 整合 → 创建文档
**输出**: "已汇总本周 3 场会议，[查看报告](URL)"

## 错误处理

| 阶段 | 错误 | 处理 | 终止? |
|------|------|------|------|
| 1 | 无法确定听记 | 列出最近 5 条供选择 | 否 |
| 2 | URL 无效 | 提示检查链接 | 是/否 |
| 2 | 摘要为空 | 尝试获取转写 | 否 |
| 4 | 创建失败 | 重试 1 次 | 是 |
| 4 | 内容过长 | 分段写入 | 否 |
