## Module A — 需求与资料获取（主 Agent 执行）

核心原则：钉钉文档优先获取内部资料，外部检索补充技术选型与最佳实践。每个步骤产出的关键信息必须写入 `draft.md`，作为 Writer Agent 的输入素材。

### A1. 需求理解

**目标**：完整理解业务目标、功能边界、非功能要求与约束条件。

**钉钉文档检索**：
```bash
# 搜索 PRD / 需求文档
dws doc search --query "{项目名称} PRD 需求文档" --format json

# 搜索产品设计文档
dws doc search --query "{项目名称} 产品设计 功能规格" --format json
```

**读取文档内容**（从搜索结果中提取 nodeId）：
```bash
# 读取找到的需求文档
dws doc read --node <DOC_ID> --format json
```

**写入 draft.md**（使用本地文件操作）：
```markdown
## 需求理解

### 业务背景
- 业务痛点：当前系统在高峰期响应缓慢，用户体验差，且维护成本持续上升
- 项目目标：重构核心链路，提升系统性能和可维护性，降低运营成本
- 关键干系人：产品负责人、技术负责人、运维团队、业务运营

### 功能需求
| 需求编号 | 需求描述 | 优先级 | 来源 |
|---------|---------|--------|------|
| REQ-001 | 用户注册登录流程优化 | P0 | PRD v1.5 |
| REQ-002 | 订单查询性能优化 | P0 | 用户投诉反馈 |
| REQ-003 | 数据导出功能 | P1 | 业务部门需求 |
| REQ-004 | 移动端适配 | P2 | 产品规划 |

### 非功能需求
- 性能：接口响应 P99 < 150ms，峰值 QPS >= 8000
- 可用性：SLA >= 99.9%，月度故障时间 < 43 分钟
- 安全：敏感字段 AES-128 加密，接口签名验证
- 扩展性：支持未来 2 年 5 倍业务量增长，支持水平扩展

### 约束条件
- 技术栈约束：基于 Go 语言开发，使用公司标准中间件
- 时间约束：需在 Q3 上线，开发周期 3 个月
- 人力约束：后端 4 人，前端 3 人，测试 2 人
- 预算约束：云资源年度预算控制在 50 万以内
```

### A2. 现有系统调研

**目标**：了解当前系统架构、技术债、可复用组件，避免重复建设。

**钉钉文档检索**：
```bash
# 搜索技术架构文档
dws doc search --query "{项目名称} 技术架构 系统设计" --format json

# 搜索架构图、部署图
dws doc search --query "{项目名称} 架构图 部署图" --format json

# 搜索相关系统接口文档
dws doc search --query "{相关系统} 接口文档 API" --format json
```

**读取文档内容**：
```bash
dws doc read --node <DOC_ID> --format json
```

**写入 draft.md**：
```markdown
## 现有系统调研

### 当前架构概述
- 架构模式：三层单体架构（表现层、业务层、数据层）
- 核心技术栈：Java 11、Spring Boot、MySQL 5.7、Redis 6.0
- 部署方式：传统虚拟机部署，使用 Nginx 负载均衡

### 可复用组件
| 组件 | 能力 | 负责团队 | 是否可直接复用 |
|------|------|---------|--------------|
| 认证网关 | 统一认证、路由转发 | 基础架构组 | 是 |
| 配置中心 | 动态配置、灰度发布 | 平台团队 | 是 |
| 日志平台 | 日志采集、检索分析 | 运维团队 | 是 |
| 监控告警 | 指标采集、异常告警 | SRE 团队 | 是 |
| 文件存储 | 对象存储、CDN 加速 | 存储团队 | 是 |

### 已知问题与技术债
- 代码耦合严重：业务逻辑与数据库访问层未分离，单元测试覆盖率低
- 缺少熔断机制：第三方服务故障会导致级联失败
- 数据一致性：分布式事务采用最终一致性，缺少补偿机制

### 现有接口依赖
- 上游：支付中心（Dubbo）、会员服务（RESTful API）
- 下游：库存服务（RocketMQ）、物流服务（gRPC）、风控服务（HTTP）
```

### A3. 技术调研

**目标**：调研行业最佳实践、架构模式和候选技术方案，为选型决策提供依据。

**web_search 查询**（多角度）：
```
- "{技术场景} 架构设计 最佳实践 2024 2025"
- "{技术A} vs {技术B} 对比 选型"
- "{业务场景} 技术方案 大厂实践"
- "{技术场景} system design architecture"
- "{中间件} 生产环境 踩坑 经验"
```

**web_fetch 深度阅读**：
```
读取技术博客/官方文档，提取架构设计、技术选型理由、性能数据、踩坑经验
```

**典型调研方向**：
- 架构模式：微服务 vs 单体、事件驱动 vs 请求驱动、CQRS、Serverless
- 数据存储：MySQL vs PostgreSQL vs TiDB、Redis vs Memcached、ES vs Meilisearch
- 消息队列：Kafka vs RocketMQ vs Pulsar
- 服务通信：gRPC vs HTTP/REST vs GraphQL
- 容器编排：Kubernetes 配置、Service Mesh

**写入 draft.md**：
```markdown
## 技术调研

### 架构模式调研
- 方案一（分层微服务）：按业务域拆分，独立部署扩展，适合复杂业务场景
- 方案二（六边形架构）：核心业务逻辑与外部依赖隔离，便于测试和维护
- 方案三（CQRS + 事件溯源）：读写分离，适合高并发写入场景

### 关键技术对比
| 维度 | 技术A | 技术B | 技术C |
|------|-------|-------|-------|
| 性能表现 | 优秀 | 良好 | 中等 |
| 学习曲线 | 平缓 | 陡峭 | 适中 |
| 社区支持 | 活跃 | 一般 | 稳定 |
| 运维复杂度 | 低 | 高 | 中 |
| 成本投入 | 低 | 高 | 中 |

### 行业参考案例
- Netflix：采用微服务架构支撑全球流媒体服务，日均处理数十亿请求（来源：Netflix Tech Blog，2024）
- Shopify：从单体迁移到微服务，部署时间从 30 分钟缩短到 2 分钟（来源：Engineering Blog，2024）
- Uber：使用事件驱动架构处理实时订单，延迟降低至毫秒级（来源：Uber Engineering，2024）
```

```markdown
## 安全与合规

### 安全要求
- 认证鉴权：OIDC + JWT，Token 有效期 1 小时，支持刷新令牌
- 数据加密：传输层 TLS 1.2，存储层 AES-128 加密敏感字段，密钥轮换周期 90 天

### 安全架构参考
- 网关层：API Gateway + WAF，统一认证和流量清洗
- 服务层：服务间双向认证，基于 RBAC 的权限控制
- 数据层：敏感数据列级加密，数据库审计日志，备份加密存储
```

```markdown
## 技术选型评估
```

### B3. 架构设计原则

> **整体方案对比**：在展开详细架构设计前，须先列出 2-3 个整体方案进行对比。例如"方案 A: 微服务架构" vs "方案 B: 模块化单体" vs "方案 C: Serverless"，从性能、成本、团队能力、演进路径等维度对比优劣，明确选择理由后再展开详细设计。

明确本方案遵循的设计原则，作为后续模块设计的指导：

```markdown
## 架构设计原则

1. **职责单一**：每个模块只负责一个业务领域，边界清晰，职责明确
2. **接口稳定**：服务间通过稳定的 API 契约通信，版本化管理，向后兼容
3. **弹性设计**：系统具备容错能力，依赖服务故障时可自动降级或熔断
4. **可验证性**：关键决策基于数据和指标，通过 A/B 测试验证假设
5. **安全优先**：安全机制贯穿设计、开发、运维全生命周期，而非事后补救
6. **全链路可见**：从请求入口到数据存储，每个环节都有日志和监控覆盖
```

## Module C — 调用 Writer Agent 生成钉钉文档

### 任务分发

主 Agent 完成数据收集和结构化分析后，调用 `task` 工具分发给 Writer Agent：

```json
{
  "description": "生成技术方案钉钉文档",
  "prompt": "基于 /home/workspace/draft.md 中的技术方案素材，生成一份结构完整的钉钉文档技术方案。\n\n要求：\n1. 严格按照技术方案文档模板结构\n2. 方案摘要（TL;DR）放在最前面，3-5 条核心设计决策\n3. 技术选型必须有评估矩阵和明确结论\n4. 性能数据标注测试条件和来源\n5. 风险清单完整，每条风险有降级方案\n6. 接口定义使用代码块\n\n文档模板见下方。",
  "subagent_type": "writer"
}
```

### Writer Agent 执行流程

```
Writer Agent 接收任务
      |
      v
1. 读取 draft.md 素材文件
      |
      v
2. 基于素材和模板生成结构化大纲
   - 输入：draft.md 内容 + 技术方案模板
   - 输出：带层级的大纲结构
      |
      v
3. 创建钉钉文档并写入内容
   步骤 3a: 创建文档
     dws doc create --name "{方案名称} 技术方案" --format json
     → 提取返回的 nodeId
   步骤 3b: 写入完整内容（追加模式）
     dws doc update --node <nodeId> --markdown "<完整方案内容>" --mode append --format json
   TRAP: markdown 内容不要以 # 一级标题开头！dws doc create --name 已经设置了文档标题，
         如果 markdown 内容再以 # 标题开头，会导致文档中出现两个重复标题。
         正文应直接从第一个章节开始（如 ## 一、项目背景与目标）
   TRAP: markdown 参数中的换行必须使用真实换行符（Unicode U+000A），
         不能使用字面量 \n，否则所有内容会渲染在同一行
   NOTE: 架构图建议使用 Markdown 表格或层级列表描述组件关系，避免 ASCII Art
         在钉钉文档中渲染异常（等宽字体不保证、对齐会错位）
      |
      v
4. 返回钉钉文档链接
   文档 URL 格式: https://alidocs.dingtalk.com/i/nodes/{nodeId}
```

### 钉钉文档操作要点

1. MUST: 创建文档时使用 `dws doc create --name "文档标题" --format json`
2. MUST: 写入内容时使用 `dws doc update --node <nodeId> --markdown "<内容>" --mode append --format json`
3. TRAP: markdown 内容禁止以 `# 标题` 开头，因为 `dws doc create --name` 已设置文档标题，重复会导致两个标题。正文从 `##` 二级标题开始
3. SHOULD: 如果内容很长，可以分多次追加写入，每次写入一个章节
4. SHOULD: 如需指定存放位置，可通过 `--folder <FOLDER_ID>` 指定文件夹
5. TRAP: `--mode overwrite` 会清空原内容后重写，谨慎使用；默认用 `--mode append`
6. TRAP: markdown 中的换行必须是真实换行符，不能是字面量 `\n`
7. NOTE: 如需精细编辑（如插入表格到特定位置），可使用 `dws doc block insert` 系列命令
