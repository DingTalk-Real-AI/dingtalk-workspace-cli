## 三、总体架构

### 3.1 架构图

架构概述：
- 接入层：渠道网关（统一接入微信/钉钉/网页/APP，协议适配）
- 智能层：NLU 引擎（意图识别、实体抽取）+ 对话管理（状态机、上下文管理）+ 知识检索（向量检索、RAG）
- 业务层：会话服务、知识库服务、工单服务、质检服务
- 数据层：MySQL（会话/用户数据）+ Redis（会话缓存）+ Milvus（向量数据库）+ Kafka（异步消息）+ ES（全文检索）
- 基础设施层：Kubernetes + 模型服务（Triton Inference Server）+ 监控告警

### 3.2 核心组件说明

| 组件 | 职责 | 技术选型 | 部署方式 |
|------|------|---------|---------|
| 渠道网关 | 统一接入、协议适配、限流 | Go + gRPC | K8s Deployment |
| NLU 服务 | 意图识别、实体抽取、槽位填充 | Python + FastAPI + Triton | K8s Deployment（GPU） |
| 对话引擎 | 对话状态管理、多轮对话逻辑 | Go + 状态机框架 | K8s Deployment |
| 知识检索服务 | 向量检索、RAG 生成 | Python + Milvus + LangChain | K8s Deployment |
| 会话服务 | 会话管理、消息存储 | Go + gRPC | K8s Deployment |
| MySQL | 会话/用户数据持久化 | MySQL 8.0 | 主从集群 |
| Milvus | 向量存储与检索 | Milvus 2.3 | 3 节点集群 |
| Redis | 会话缓存、分布式锁 | Redis 7.0 Cluster | 3 主 3 从 |
| Kafka | 异步消息、事件驱动 | Kafka 3.x | 3 Broker 集群 |

### 3.3 数据流

描述核心业务场景下的数据流转路径：

1. 用户发送消息 -> 渠道网关 -> 协议适配 -> 对话引擎
2. 对话引擎 -> 查 Redis 会话上下文 -> 调用 NLU 服务识别意图和实体
3. NLU 服务 -> 意图/实体结果 -> 对话引擎 -> 根据对话状态决定下一步
4. 如需知识库问答 -> 调用知识检索服务 -> Milvus 向量检索 -> RAG 生成答案
5. 对话引擎 -> 生成回复 -> 渠道网关 -> 返回用户
6. 异步流程：对话引擎 -> 发 Kafka 消息 -> 会话服务持久化 -> 质检服务分析

## 四、关键模块设计

### 4.1 模块A: 对话管理模块

#### 职责
- 负责对话状态管理、多轮对话流程控制、上下文维护
- 不负责意图识别（由 NLU 模块处理）和知识检索（由知识检索模块处理）

#### 接口设计

POST /api/v1/conversation/message
Request:
  session_id: string, 必填, 会话ID
  user_id: string, 必填, 用户ID
  channel: string, 必填, 渠道类型（web/app/wechat/dingtalk）
  message:
    content: string, 必填, 用户消息内容
    message_type: string, 必填, 消息类型（text/image/file）
    timestamp: int64, 可选, 消息时间戳

Response:
  code: 0
  message: "success"
  data:
    session_id: "string"
    reply:
      content: "string", 回复内容
      message_type: "string", 消息类型
      intent: "string", 识别的意图
      confidence: float, 意图置信度
      need_transfer: boolean, 是否需要转人工
    next_actions:
      - action: "collect_slot", 收集槽位
        slot_name: "string", 槽位名称
        prompt: "string", 提示语
      - action: "search_kb", 检索知识库
        query: "string", 检索查询

错误码:
- 20001: 会话不存在或已过期
- 20002: 消息格式错误
- 20003: 用户已被禁用
- 20004: 服务限流

| 字段 | 类型 | 必填 | 索引 | 说明 |

| user_id | VARCHAR(64) | 是 | IDX | 用户ID |
| channel | VARCHAR(32) | 是 | IDX | 渠道类型 |
| status | TINYINT | 是 | IDX | 状态: 0-活跃 1-等待人工 2-已关闭 |
| current_intent | VARCHAR(64) | 否 | | 当前意图 |
| context_json | JSON | 否 | | 对话上下文（槽位、历史消息等） |
| last_active_at | DATETIME | 是 | IDX | 最后活跃时间 |
| created_at | DATETIME | 是 | IDX | 创建时间 |
| updated_at | DATETIME | 是 | | 更新时间 |

表名: t_conversation_message

| 字段 | 类型 | 必填 | 索引 | 说明 |
|------|------|------|------|------|
| id | BIGINT | 是 | PK | 自增主键 |
| message_id | VARCHAR(64) | 是 | UNI | 消息唯一标识 |

#### 关键逻辑
- 并发控制：使用 Redis 分布式锁，锁粒度为 session_id，防止同一会话的并发请求导致状态混乱
- 状态机：活跃 -> 等待人工 -> 人工处理 -> 活跃/已关闭
- 幂等设计：基于 message_id 做幂等校验，避免重复处理
- 上下文管理：使用 Redis 缓存会话上下文，TTL 30 分钟，过期后从 MySQL 加载

## 五、技术选型

### 5.1 选型决策

| 决策点 | 方案A | 方案B | 方案C | 结论 |
|--------|-------|-------|-------|------|
| 向量数据库 | Milvus 2.3 | Qdrant 1.7 | Weaviate 1.24 | Milvus 2.3 |
| NLU 框架 | Rasa 3.x | Dialogflow ES | 自研 | 自研 |
| Embedding 模型 | BGE-M3 | text-embedding-ada-002 | m3e-base | BGE-M3 |
| 对话引擎 | LangChain | 自研状态机 | Microsoft Bot Framework | 自研状态机 |
| 模型推理 | Triton Inference Server | TorchServe | vLLM | Triton Inference Server |
| 消息队列 | Kafka | RocketMQ | Pulsar | Kafka |
| 服务框架 | Go + gRPC | Python + FastAPI | Java + Spring Boot | Go + gRPC |

### 5.2 选型理由

#### 向量数据库: Milvus 2.3
- **选择原因**：开源社区活跃，支持多种索引类型（HNSW/IVF），性能优异，团队有 1 年生产使用经验
- **放弃 Qdrant 的原因**：虽然易用性好，但在大规模数据场景下性能不如 Milvus，且生态集成度较低
- **放弃 Weaviate 的原因**：资源占用较高，运维复杂度大，且缺乏大规模生产验证
- **迁移预案**：当向量数据超过 5000 万条时，考虑分片或迁移至云原生向量数据库

#### NLU 框架: 自研
- **选择原因**：业务场景特殊，需要深度定制意图识别逻辑（如多意图混合、槽位继承），自研可以更好地控制推理性能和模型热更新
- **放弃 Rasa 的原因**：虽然功能完整，但框架较重，推理延迟较高，且定制化成本不低
- **放弃 Dialogflow 的原因**：SaaS 服务，数据隐私风险，且无法满足定制化需求

#### Embedding 模型: BGE-M3
- **选择原因**：开源模型，中文表现优异，支持多语言，支持长文本（8192 tokens），推理速度快（GPU 下 < 10ms）
- **放弃 text-embedding-ada-002 的原因**：闭源模型，成本高，且中文表现不如 BGE-M3
- **放弃 m3e-base 的原因**：虽然中文表现好，但不支持长文本，在文档问答场景下受限

#### 模型推理: Triton Inference Server
- **选择原因**：支持多种框架（PyTorch/TensorFlow/ONNX），支持动态批处理和 GPU 加速，与 Kubernetes 集成良好
- **放弃 TorchServe 的原因**：仅支持 PyTorch 生态，扩展性较差
- **放弃 vLLM 的原因**：虽然推理性能优异，但仅支持 LLM，不支持 Embedding 模型

#### 消息队列: Kafka
- **选择原因**：高吞吐量满足需求，与公司数据平台生态一致，团队经验丰富，支持消息回溯
- **放弃 RocketMQ 的原因**：虽然延迟更低，但团队无运维经验，且生态集成度不如 Kafka

### 6.3 压测方案

- **压测工具**：Locust / k6 / Gatling
- **压测环境**：独立压测集群，配置与生产等比缩放（包括 GPU 实例）
- **压测场景**：
  1. 单接口性能基线测试（意图识别、知识检索、对话响应）
  2. 混合场景（查询/对话比 7:3）压测
  3. 峰值流量（2.5 倍日常）持续 30 分钟
  4. 长时间稳定性压测（24 小时，1.2 倍日常流量）
  5. 故障注入（杀节点、GPU 故障、网络延迟、Milvus 故障）下的性能表现
- **通过标准**：P99 达标 + 错误率 < 0.5% + 无 OOM / 无长时间 GC + 意图识别准确率无明显下降

## 七、安全与合规

- **认证鉴权**：API Gateway 统一鉴权，OAuth 2.0 + JWT
- **数据安全**：敏感数据（手机号、身份证号等）加密存储，日志自动脱敏
- **传输安全**：全链路 TLS 1.3，服务间 mTLS
- **接口防护**：限流（令牌桶）、防重放、CSRF 防护、XSS 过滤
- **审计日志**：所有写操作记录审计日志，保留 90 天
- **合规**：满足公司安全基线要求，上线前通过安全审核

## 八、风险与降级

### 8.1 风险清单

| 风险 | 概率 | 影响 | 降级方案 | 监控指标 | 告警阈值 |
|------|------|------|---------|---------|---------|
| NLU 模型漂移 | 中 | 严重 | 回滚到上一版本 + A/B 测试验证 | 意图识别准确率、置信度分布 | 准确率 < 85% |
| 知识库更新延迟 | 高 | 中 | 使用旧版本知识库 + 告警 | 知识库同步延迟、向量索引状态 | 同步延迟 > 10min |
| 高并发会话堆积 | 中 | 严重 | 限流 + 自动扩容 + 降级到简单回复 | 并发会话数、队列长度 | 队列 > 1000 |
| Milvus 查询超时 | 低 | 严重 | 降级到全文检索（ES）+ 告警 | Milvus 查询延迟、错误率 | P99 > 2s |
| GPU 资源不足 | 中 | 中 | 动态扩容 + 降级到 CPU 推理 | GPU 利用率、推理队列长度 | GPU > 90% |
| 对话上下文丢失 | 低 | 严重 | Redis 持久化 + MySQL 备份 | Redis 缓存命中率、会话恢复成功率 | 命中率 < 95% |
| Embedding 模型超时 | 中 | 中 | 重试 + 降级到关键词匹配 | Embedding 推理延迟、错误率 | 错误率 > 1% |
| 第三方渠道 API 变更 | 低 | 中 | 版本兼容层 + 快速适配 | 第三方 API 调用成功率 | 成功率 < 99% |

### 8.2 灰度策略

- **灰度维度**：按用户 ID 尾号/渠道/业务线分组
- **灰度阶段**：
  1. 内部测试（全员）-> 5% 用户（某业务线）-> 20% 用户（多业务线）-> 50% 用户 -> 全量
  2. 每个阶段至少观察 48 小时
  3. 核心指标（意图识别准确率、会话解决率、P99 延迟）无异常后推进下一阶段
- **回滚条件**：错误率 > 1% 或 P99 > 2s 或意图识别准确率下降 > 3% 自动回滚

### 9.2 测试策略

- **单元测试**：核心模块覆盖率 >= 75%，提交时自动运行
- **集成测试**：核心链路端到端覆盖，每日自动回归
- **模型测试**：意图识别准确率 >= 92%，Embedding 召回率 >= 95%
- **性能测试**：上线前压测，达到 2 倍预期峰值流量
- **安全测试**：上线前安全扫描，修复所有高危漏洞
- **灰度验证**：按灰度策略逐步放量，观察核心指标（准确率、解决率、延迟）

## 十、监控与运维

- **指标监控**：QPS、延迟（P50/P99/P999）、错误率、CPU/内存/磁盘、GPU 利用率、意图识别准确率、会话解决率
- **日志监控**：ERROR 日志实时告警，核心链路日志全量采集，对话日志保留 90 天
- **链路追踪**：OpenTelemetry 全链路追踪，支持请求级别排查（用户消息 -> NLU -> 对话引擎 -> 知识检索）
- **告警规则**：
  - 错误率 > 0.5%：P1 告警，自动通知值班人
  - P99 > 2s：P2 告警，通知负责人
  - 意图识别准确率 < 85%：P1 告警，通知算法团队
  - 服务不可用：P0 告警，电话通知 + 自动拉群
- **Runbook**：常见故障处理手册，降低 MTTR
