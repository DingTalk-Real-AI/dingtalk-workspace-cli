# 常见失败模式与错误处理

## 常见失败模式与修复

| 失败模式 | 典型表现 | 根本原因 | 修复方法 |
|---------|---------|---------|---------|
| 需求理解偏差 | 方案实现的功能与业务目标不符，技术评审时被质疑"这个功能的价值是什么" | 未能深入查阅产品需求文档，仅依赖用户简短描述就启动设计 | 必须执行 Module A1：先通过 `dws doc search` 检索 PRD，再用 `dws doc read` 提取业务背景，需求列表需与 PRD 逐一核对 |
| 技术选型无依据 | 选型结果仅给出"采用 XX"，缺乏对比分析，评审时被问"为何不选 YY"时无法回应 | 省略了选型评估矩阵，直接依赖过往经验或个人偏好 | 必须执行 Module B2：每个核心技术决策需包含至少 2 个候选方案的多维对比，清晰说明未采纳其他方案的原因 |
| 缺少性能估算 | 方案未包含 QPS 指标、容量规划和资源预算，上线后才发现无法承载实际流量 | 未建立流量模型，未检索性能基准数据 | 必须执行 Module A4 和 B5：需包含流量模型（DAU 转换 QPS）、基准测试数据（注明来源）、资源估算表和扩展方案 |
| 风险遗漏 | 上线后遭遇数据库故障、缓存崩溃等异常，方案中完全未预见 | 仅关注"正常链路"设计，未系统化考虑异常场景 | 必须执行 Module B6：风险清单至少 5 项，每项需包含概率评估、影响评估和降级方案；评审前逐项演练降级预案 |
| 方案过于理想化 | 架构追求"极致"，引入众多高级组件，但团队仅有 3 名成员 | 忽视团队能力和资源限制，过度设计 | 在 Module B3 架构原则中加入"渐进式演进"原则；选型矩阵中"团队经验"和"运维成本"权重不低于 15% |
| 模块设计粗糙 | 模块仅有名称和简短描述，缺失接口定义、数据模型和核心逻辑 | Module B4 执行不到位，模块设计停留在架构图层级 | 强制要求每个 P0 模块必须包含：职责边界、接口设计、数据模型、核心逻辑 |
| 钉钉文档格式错乱 | 所有内容显示在同一行，标题、段落、表格全部混杂 | markdown 参数使用了字面量 `\n` 而非真实换行符 | 确保 markdown 内容中的换行使用真实换行符（Unicode U+000A），程序构造时正确处理转义 |
| 风险表缺少可观测性 | 风险清单有降级方案但没有监控指标和告警阈值，无法自动触发降级，只能靠人工发现 | 只关注了"怎么降级"而忽略了"怎么发现需要降级" | 风险清单每项必须包含监控指标和告警阈值，确保降级策略可被自动触发 |
| 选型覆盖不全 | 只对最核心的一个组件做了选型分析（如只选了框架），存储、缓存、队列等基础设施直接使用未经论证 | Module B2 执行不充分，误以为只需对"最重要的"做选型 | 每个基础设施组件都须有选型分析，理由明确的可简化为一段说明，但不能省略 |
| 模块设计混入实现代码 | 模块设计章节中出现大段类定义、函数体、伪代码，文档像代码仓库而非架构设计 | 未区分"设计"和"实现"的边界，把编码细节提前写入方案 | 模块设计聚焦职责边界、接口契约和数据模型；代码示例统一放在附录 |

## 错误处理

| 错误 | 原因 | AI 应该怎么做 |
|------|------|--------------|
| dws doc search 无结果 | 检索关键词不匹配或文档不存在 | 使用更宽泛的关键词重新尝试，或请求用户提供文档链接 |
| dws doc read 返回 403 | 用户缺少文档访问权限 | 提示用户"您没有该文档的访问权限，请确认权限后重试" |
| dws doc create 失败 | 可能是目录权限或参数配置错误 | 添加 `--verbose` 参数重试一次，若仍失败则报告完整错误详情 |
| dws doc update 内容格式错乱 | markdown 换行符使用了字面量 `\n` | 确保使用真实换行符重新写入 |
| web_search 无相关结果 | 搜索关键词过于专业或冷门 | 使用英文关键词或更通用的表述重新尝试 |
